Use Case 01 · Collaborative AI

CoWiki: Collaborative Intelligence Without Merge Conflicts

Multiple AI agents contribute simultaneously to a shared knowledge base. No write locks, no coordinator, no conflicts. AION's semantic graph finds the truth through structure.

Primitives7 MCP tools Agents3 concurrent Latent Edges47 inferred OutputMerged knowledge graph

The Problem

Traditional wikis require serialized writes. A central lock arbitrates who can write, one author at a time. When multi-agent systems try to contribute simultaneously, they collide: optimistic concurrency fails on overlapping sections, last-write-wins discards work, and merge conflicts demand a human or coordinator to resolve.

The serialization assumption is a relic of single-author systems. An AI agent that has synthesized a section on Byzantine fault tolerance should not be blocked waiting for a sibling agent to finish its section on Paxos consensus. These are independent contributions to a shared knowledge structure — the relationship between them is a graph problem, not a scheduling problem.

CoWiki solves this by treating each agent's contribution as a signed, content-addressed capsule written independently to its own .aion file. No lock is acquired. No coordinator is consulted. Once written, AION's semantic graph merges contributions by discovering structural relationships between them. The graph finds the truth; the agents just write.

Session Setup

Three agents are assigned to a collaborative article on distributed consensus algorithms. Each agent operates independently — no shared state, no coordination channel, no awareness of what the other agents are writing. They write in parallel; AION merges their work after the fact.

agent-alpha agent-beta agent-gamma

Each agent is assigned a domain: Byzantine fault tolerance (alpha), Paxos consensus (beta), and Raft leader election (gamma). Their contributions are independent — no agent reads what the others have written before writing its own capsule. When all three have finished, the substrate sees three isolated nodes. The merge happens in the graph layer.

Step 1 — Node Creation

Each agent calls aion_annotate to write its contribution as a signed capsule. AION assigns a BLAKE3 content address — the canonical node identifier. No two agents write to the same file. There are no locks because there is nothing to lock against: independent capsules share no mutable state.

Step 2 — Semantic Linking

Once nodes exist, the graph layer draws associations between related concepts. aion_link creates directed, typed edges. In this session an orchestrating pass encodes the known conceptual relationships: both alpha and gamma require quorum; Paxos generalizes the ideas Raft makes concrete; all three sections discuss safety under failure.

Step 02 · Semantic Linking — aion_link × 3

Three explicit edges form the initial skeleton of the merged knowledge graph. The links are typed and labelled — the substrate knows not just that two nodes are related, but why.

Step 3 — Context Retrieval

A downstream consumer — a reader agent, a synthesis pass, or a search interface — issues a query against the merged graph. aion_query performs semantic retrieval across all contributing agents' nodes simultaneously, returning ranked hits weighted by relevance. The query engine is unaware of which agent authored which node: it sees one coherent corpus.

Step 03 · Semantic Retrieval — aion_query "fault tolerance"
aion_query
alpha-node · Byzantine fault tolerance 0.94
gamma-node · Raft leader election 0.71
beta-node · Paxos consensus 0.63

All three agent contributions are discovered and ranked in a single query pass. The merged graph is transparent to the consumer — it behaves identically to a graph written by a single coordinated author.

Step 4 — Dream Pass

The explicit links encode what the linking agent knew to encode. The latent structure of the knowledge graph contains more. aion_dream propagates spreading activation from all known nodes, scores every candidate edge the explicit link set does not yet cover, and materializes those scoring above the threshold. Relationships emerge from structural proximity alone — no agent needed to name them.

Step 04 · Latent Edge Discovery — aion_dream threshold=0.28
aion_dream

47 latent edges inferred above the 0.28 threshold. Decisive findings:

alpha-node ↔ gamma-node
Jaccard 0.41 both discuss quorum majority
beta-node ↔ alpha-node
Jaccard 0.33 both cite Lamport's 1978 paper

The dream pass surfaces relationships no agent explicitly encoded. The alpha–gamma quorum connection and the beta–alpha Lamport citation were latent in the activation topology, discovered without any agent labelling them. This is the substrate augmenting agent knowledge rather than merely storing it.

Step 5 — Journal Episode

The full merge sequence is recorded as a temporal episode. aion_journal_append writes an ordered, cryptographically chained log of every event in the session — individual agent writes, the dream pass, and the context merge. This provides a reproducible, auditable record of exactly how the merged knowledge base was constructed.

Each event references the BLAKE3 address of the node it acted on. A future reader can reconstruct the exact state of the knowledge base at any point in the session — including the intermediate state before the dream pass ran.

Step 6 — Snapshot Seal

Once the merge is complete and the dream pass has densified the graph, the merged knowledge base is sealed. aion_snapshot computes a BLAKE3-chained digest of the full node and edge set, signs it with Ed25519, and returns an immutable truth checkpoint. Any subsequent mutation — accidental or adversarial — will produce a different digest. The seal is the proof of what the graph contained at the moment of merge.

Key Findings

  • 01 Zero write conflicts across three concurrent agents. Each agent wrote to its own capsule with no lock acquisition. Concurrency is structurally guaranteed — there is no shared mutable state to conflict on.
  • 02 Semantic merge without a coordinator. The graph layer merged three independent contributions through typed links and spreading activation. No central process negotiated the merge. The structure did.
  • 03 Dream pass discovered two relationships no agent explicitly encoded. The alpha–gamma quorum connection and the beta–alpha Lamport citation were latent in the activation topology, surfaced without any agent labelling them.
  • 04 Full session is reproducible and auditable. The journal records every event in order. The snapshot seals the final state. Any party can verify that the merged graph is exactly what was produced by this session and nothing else.
  • 05 Retrieval is agent-agnostic. The query engine returned ranked results without distinguishing which agent authored which node. The merged knowledge base behaves as a single coherent corpus from the consumer's perspective.
Core Insight

No locks, no conflicts, no coordinator. Each agent writes freely to its own signed capsule; the graph finds the truth through structure. AION makes multi-agent collaboration feel like git — distributed, signed, mergeable — but at the semantic level rather than the line-diff level. Two agents can write overlapping sections on the same topic and the substrate will surface their relationship as a graph property rather than a merge conflict requiring human resolution.

Implementation

Paul Shomo's original CoWiki concept is prototyped in cowiki-rs, a Rust implementation that wires the multi-agent write pattern to an AION-compatible substrate. The prototype demonstrates that the lock-free write model is not merely theoretical — three concurrent threads write independent capsules, the graph layer merges them, and the dream pass runs as a post-write reconciliation step that densifies what the explicit links did not fully express.

The full technical analysis of the design — including the formal argument for why content-addressed capsules eliminate write conflicts by construction — is documented in the original technical writeup.
