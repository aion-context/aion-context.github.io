Use Case 02 · Security Intelligence

Operation NightShift

A cross-referenced vulnerability disclosure and live network telemetry triggered a graph-driven security investigation — reconstructing a 3-hop attack chain, quantifying blast radius, and sealing the post-mortem evidence in under four hours.

Primitives7 MCP tools Attack chain3 hops Blast radius7 nodes OutputSealed post-mortem

Incident Context

At 02:17 local time, a network intrusion detection system flagged anomalous outbound traffic from a production service cluster. The pattern was distinctive: small, regularly-timed packets to an unfamiliar external endpoint, consistent with a command-and-control beacon. The on-call engineer's first question was not whether an incident had occurred — that much was clear — but how. What was the attack chain? Where did the adversary enter, and what could they reach from there?

The answer was already partially documented. Six weeks earlier, a CoWiki page had been published to the internal knowledge base: a detailed disclosure of a memory safety vulnerability in a binary that ran on the affected cluster. The disclosure included a proof-of-concept exploit payload hash, a list of affected versions, and a recommendation for remediation. The patch had been scheduled but not yet deployed.

Both the vulnerability disclosure and the live incident data were in AION. The investigation began by connecting them.

Node Inventory

The investigation ingested four primary node categories. The CVE disclosure node had already been created by the CoWiki authoring flow — it was a content-addressed capsule signed by the security researcher who wrote the post. The binary node, network telemetry nodes, and exploit payload node were ingested fresh at incident time via aion_annotate.

Step 1 · Attack Chain: aion_path

The first question was structural: what is the shortest path from the CVE disclosure — the publicly available knowledge that enabled this attack — to the compromised service? aion_path was called with a causal constraint, traversing only edges typed as causal, exploits, or affects.

Step 2 · Blast Radius: aion_impact

With the entry point confirmed, the investigation needed to quantify exposure. From internal-service-A — the beachhead host — what could an adversary with a foothold reach, and at what depth? aion_impact traversed forward from the compromised service with a 3-hop constraint.

aion_impact — internal-service-A, forward, 3 hops
aion_impact
  • D1internal-service-B — directly reachable via service mesh; shares authentication context
  • D1lateral-host-01 — adjacent compute node; no network segmentation boundary
  • D2network-telemetry-02 — telemetry collector reachable from lateral-host-01

  • D3logging-aggregator — reachable via telemetry path; log injection surface
  • D3config-management-node — reached via lateral host chain
  • D3data-pipeline-ingress — downstream of service-B; potential exfiltration path

7 nodes reachable within 3 hops. The blast radius extended significantly beyond the initial compromise point, revealing that insufficient network segmentation allowed lateral movement across both compute and telemetry planes.

Step 3 · Cluster Detection: aion_community

The impact analysis identified seven reachable nodes — but were they a coherent cluster, or did they span multiple independent sub-networks with different risk profiles? aion_community partitioned the full node set.

aion_community — Infrastructure Partition
aion_community

The algorithm identified a single primary cluster containing the compromised service, all reachable downstream nodes, and the network telemetry collectors. The c2-beacon-endpoint and exploit-payload nodes formed a distinct external cluster with a single inbound edge — the connection through which the exploit was delivered.

Cluster partitioning answered a critical containment question: which nodes could be isolated together without severing unrelated infrastructure dependencies. The primary cluster boundary defined the containment perimeter.

Community identifiers omitted from this operational summary.

Containment decision: The community boundary identified by aion_community was used directly to scope the emergency network isolation. Every node in the primary cluster was isolated before the end of the incident response window.

Step 4 · Historical Pattern: aion_recall

Was this the first time this vulnerability class had appeared in the environment? aion_recall was seeded from the cve-disclosure and exploit-payload nodes, propagating activation across the full historical graph — including nodes from previous investigation contexts that had not been part of the initial ingestion for this incident.

aion_recall — Seeds: cve-disclosure + exploit-payload
aion_recall

Spreading activation surfaced three related incidents from the historical graph:

  • 01Q3 2024 staging incident — a binary from the same vendor exhibited anomalous behavior in a staging environment; closed as a misconfiguration. Activation score: 0.081. The exploit-payload hash overlaps with tooling observed in that incident.
  • 02CVE sibling disclosure — a related vulnerability in the same binary family, disclosed eight months prior, was patched but not fully deployed to all cluster nodes. Activation score: 0.064. At least one lateral host ran the unpatched version.
  • 03C2 infrastructure overlap — the beacon endpoint IP appeared in a threat intelligence node from a previous investigation context, associated with a known initial access broker. Activation score: 0.042.

None of these connections were visible to the on-call engineer at incident time. They emerged from the graph's implicit structure, surfaced by spreading activation without any manual cross-referencing.

Step 5 · Incident Timeline: aion_journal_append

As the investigation progressed, each significant event was appended to the incident journal. The journal provides an ordered temporal record — not a log file, but a structured, content-addressed sequence of episodes, each linked to the nodes it references.

T+00:14 CVE cross-reference identified. CoWiki disclosure node matched to exploit payload hash. aion_link called to connect the two contexts.
T+00:31 Attack chain reconstructed. 3-hop path confirmed via aion_path. On-call engineer briefed; incident declared P1.
T+00:52 Blast radius quantified. 7 nodes reachable within 3 hops. Containment perimeter defined by community detection output.

T+03:44 Post-mortem draft complete. All findings, node links, and timeline entries recorded. Snapshot requested.

Step 6 · Evidence Seal: aion_snapshot

At the conclusion of the investigation, the full incident context — every node, edge, and journal entry — was sealed with aion_snapshot. The snapshot digest and Ed25519 signature were attached to the post-mortem document and filed with the security operations record. The sealed state is the authoritative record of what the investigation knew at close, and when it knew it.

Key Findings

  • 01 Attack chain reconstructed in 31 minutes. The 3-hop causal path from CVE disclosure to compromised service was identified via aion_path without manual log correlation. The connection to the CoWiki disclosure was made automatically because both nodes lived in the same graph.
  • 02 Blast radius: 7 downstream nodes within 3 hops. The impact analysis revealed that the compromised host had reachable access to credential stores, data pipelines, and logging infrastructure — a significantly larger exposure than initial intuition suggested.
  • 03 Containment perimeter defined algorithmically. Community detection identified the cluster boundary without manual grouping. The isolation decision followed directly from the graph output.
  • 04 Three related historical incidents surfaced. Spreading activation revealed a Q3 2024 staging anomaly, an unpatched sibling CVE, and a C2 infrastructure overlap — none visible without cross-graph query.
  • 05 Tamper-evident post-mortem sealed for filing. The BLAKE3 + Ed25519 snapshot provides a complete, non-repudiable record of the investigation state at close — suitable for regulatory submission, legal hold, and future reference.
